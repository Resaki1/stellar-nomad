# web:nubis-decima

## SUMMARY
The Nubis lineage (Andrew Schneider, Guerrilla/Decima) evolved through four public iterations, and the primary slide decks for all four were fetched and text-extracted in full, so the findings below are quoted from primary sources, not summaries. The arc is: (1) SIGGRAPH 2015 (Horizon Zero Dawn prototype) — a single RGB "weather map" (R=coverage, G=precipitation, B=cloud type, all 0-1) drives a raymarcher whose vertical shape comes from exactly 3 analytic height-gradient presets (stratus/stratocumulus/cumulus) blended by the type value; (2) SIGGRAPH 2017 (shipped Nubis) — the map is reframed as an authored "cloud map" over ~100 km² (R=Perlin coverage for connected formations, G=Perlin-Worley coverage for isolated islands, B=type where 0=stratus, 0.5=stratocumulus, 1=cumulus), the height presets become remap() gradients whose in/out points are continuously interpolated by type (this is the canonical no-binary-border mechanism), and anvils are a pow() on coverage near the layer top gated by a per-weather-state anvil_bias; (3) SIGGRAPH 2022 "Nubis, Evolved" (Horizon Forbidden West) — the single type channel splits into independent Bottom Type and Top Type channels indexing vertical-profile lookup textures (LUTs), plus Min/Max Height channels, in a stack of five 16km×16km 2D "Nubis Data Fields" (NDFs); a separate 2.5D model (coverage+type) renders the cirrus sub-layer, an Envelope model (min/max height, type, density) renders fly-through orographic clouds, and superstorms/anvils are authored as influence-NDF overrides with concentric rotation rings; (4) SIGGRAPH 2023 "Nubis3 / Nubis Cubed" (Burning Shores) — clouds the player flies through are no longer generated from a 2D weather map at all: artists build voxel cloudscapes in Houdini ("frankenclouding" fluid-sim pieces via the Atlas tools) baked into 3D NVDFs (512×512×64 over 4km×4km×500m: Dimensional Profile derived from an SDF, Detail Type, Density Scale, plus a separately-stored BC1-packed SDF for sphere-traced step placement), up-rezzed at sample time from 8m to ~0.5m precision with new Alligator/Curly-Alligator detail noises and a "twice-folded" ultra-high-frequency noise inside 150m of the camera.

The through-line most relevant to Stellar Nomad's redesign: in every iteration, cloud "type" is a continuous scalar (or two scalars in 2022) that interpolates the PARAMETERS of a vertical density profile (remap in/out points or LUT rows), never a discrete branch — so mixed skies emerge from spatially-varying type/coverage fields rather than from blending finished cloud looks, which is exactly what eliminates binary borders. Vertical profile and detail character are deliberately coupled to the same axes: detail noise blends wispy→billowy by type (and by height within a cloud in 2022's envelope model, and by depth-into-cloud via the dimensional profile in Nubis3), and lighting probabilities (ambient, multiple-scatter, in-scatter darkening) are all derived from the same dimensional-profile field, so one authored field feeds density, detail selection, and lighting. The consumption path is uniform: sample 2D (or 3D) authored fields → build dimensional profile → erode with type-selected noise composite — which is the property Stellar Nomad needs for R8 (far-field overlay and volumetric consuming the same maps) and R4 (procedural planets can generate the same field stack).

For fly-through (R6-adjacent), Nubis3's lessons are: distance-based detail addition (fold existing noise channels to synthesize higher octaves near camera, blend in under 150m, plus density sharpening pow() to fight undersampling), SDF-driven step placement max()ed with a sqrt(distance)-scaled adaptive step, animated-near/static-far jitter, decoupled precomputed lighting (a coarse 256×256×32 summed-density-toward-sun grid amortized over 8 frames replacing all but the first 2 light-march samples, -40% frame cost and enabling long-distance intercloud shadows), and a 2-pass resolution split (<200m at quarter-res, >200m at half-res, ~-50%). Weather-map-driven evolution was sacrificed in Nubis3 (pseudomotion only, wind-scrolled noise under static structure); the sky-level vertical-profile system in HFW retained NDF stacks selected/blended by weather states.

## 2015 (HZD prototype) weather map: R=coverage, G=precipitation, B=type; simulation-driven; cumulonimbus forced at 70% coverage by precipitation [confirmed]
SIGGRAPH 2015 deck, pages 41-48 of extracted text: 'Red is coverage, Green is precipitation and blue is cloud type. The weather system modulates these channels with a simulation that progresses during gameplay', with bias controls for art direction. Coverage and type are 0-1 values sampled per-column; 'The precipitation signal transitions the map from whatever it is to cumulonimbus clouds at 70% coverage' and also drives rain VFX and artificially increased light absorption (darker rain clouds). Distance shaping: cloudscape drawn within a 35,000 m radius; beyond 15,000 m from the player the system biases toward cumulus at ~50% coverage so horizons always look 'epic' and poke above mountains. Default condition is a cumulus+stratus mix (where R is high and B low you get stratus). For cutscenes the sim output is overridden with hand-painted textures. Map resolution is NOT stated in the slides (a 512x512 figure circulates from the GPU Pro 7 book chapter but is unconfirmed here). Animation: the whole system is animated along a wind direction; in 2015 both coverage signal and noise move.

## 2015 vertical profiles: 3 analytic height presets blended by type; altitude bands 1500-4000m volumetric + >4000m 2D textures [confirmed]
Instead of one height gradient, '3 mathematical presets that represent the major low altitude cloud types' (stratus, stratocumulus, cumulus) are blended between at the sample position using the type value — i.e. type is already a continuous interpolation axis, not a switch. Density is reduced at cloud bottoms ('bottoms will be whispy... density increases over altitude'). Sphere-shell march: low-altitude strato-class clouds are volumetric between 1500 m and 4000 m; alto/cirro class above 4000 m are cheap scrolling 2D textures ('collection of the various types of cirrus and alto clouds tiling and scrolling at different speeds and directions'). March budget: 64 potential samples at zenith to 128 at horizon; 6 cone samples toward sun (1 long-range for distant shadows); light samples switch to the cheap (base-noise-only) sampler once accumulated alpha > 0.3 (2x faster); temporal reprojection updates 1/16 pixels per 4x4 block at quarter res, ~10x speedup, ~2 ms on PS4.
ANCHORS: https://advances.realtimerendering.com/s2015/The%20Real-time%20Volumetric%20Cloudscapes%20of%20Horizon%20-%20Zero%20Dawn%20-%20ARTR.pdf

## 2015 noise stack: 128^3 Perlin-Worley+3 Worley base, 32^3 Worley detail, 128^2 curl; detail as edge erosion, inverted Worley at bases for wisps [confirmed]
Texture budget: 3D texture #1 = 128^3, 4 channels (ch1 Perlin-Worley = Perlin dilated by inverted Worley; ch2-4 Worley at increasing frequencies) defines the base cloud shape. 3D texture #2 = 32^3, 3 channels of higher-frequency Worley — subtracted (eroded) only at the edges of the base shape, so cores keep density. 2D texture = 128^2, 3-channel curl noise used to distort the detail-noise sample position for turbulence. Wispy vs billowy per-position rule: 'If you invert the Worley noise at the base of the clouds you get some nice whispy shapes' — i.e. billow (packed round Worley) near tops, inverted/wispy near bases. Because detail only erodes edges, the marcher can take cheap base-noise-only samples until an isosurface is hit, then step back and do full samples — the fetch-per-voxel currency argument the whole lineage optimizes.
ANCHORS: https://advances.realtimerendering.com/s2015/The%20Real-time%20Volumetric%20Cloudscapes%20of%20Horizon%20-%20Zero%20Dawn%20-%20ARTR.pdf

## 2017 (shipped Nubis) cloud map: RGB over ~100 km²; R=Perlin coverage (connected), G=Perlin-Worley coverage (islands), B=type with 0=stratus/0.5=stratocu/1=cumulus [confirmed]
SIGGRAPH 2017 deck pages 53-59: 'The cloud map is an RGB buffer which represents a coverage area of about 100 square kilometers. The red and green channels represent cloud coverage' — a low-frequency noise modulates where air masses overlap, then higher-frequency noises form cloud footprints: 'Perlin noise is used in the red channel, while a Perlin-Worley noise is used in the green channel. Either noise is used in the density sampler depending on if the artist wants connected cloud formations (Perlin) or more isolated island-like shapes with connective tissue (Perlin-Worley)'. 'Another low frequency noise is used to define the modulation in cloud type. This goes into the blue channel, with a value of Zero being stratus, 1 being cumulus or value of .5 indicates stratocumulus'. Notably precipitation is no longer a map channel; per-sky-state scalars moved into a 'weather state' object: skew, anvil inflation (anvil_bias), movement speed, movement direction, cloud density — 'treated same at every sample, so they didn't need to be stored in the Cloud Map'. Weather scheduler: ~3 basic presets per zone cycled by chance (gated by time of day, e.g. no cumulus mornings), 10 regional zones + 70 special zones; state switches are a simple lerp of the current state toward the next over 10-30 seconds; crossing a zone boundary swaps the next scheduled state and starts the blend immediately. Cutscene skies are hand-edited sim outputs / composited 'Frankencloudscape' maps; art holes are literally painted black into the map.

## 2017 type->profile mechanism: remap() gradients whose in/out points are interpolated BY type (the canonical smooth-transition trick), plus exact erosion formulas [confirmed]
The height probability for a type is two multiplied remaps, e.g. stratus = remap(height, 0.0, 0.1, 0.0, 1.0) * remap(height, 0.2, 0.3, 1.0, 0.0) (remap(v, omin, omax, nmin, nmax) = nmin + ((v-omin)/(omax-omin))*(nmax-nmin)). 'If we wish to describe all of the stages of a cloud in the minus layer, we can simply adjust the in and out points of our remap function... we can order a set of these adjustments according to cloud type... Type, can then serve as the value which modifies the in and out points of our remap functions.' I.e. type does NOT crossfade between three finished profiles — it continuously slides the gradient endpoints along the stratus->stratocumulus->cumulus ordering, so every intermediate type is itself a plausible profile and borders in the type field become gradual morphs (directly the fix for Stellar Nomad's binary border). Core density formulas: perlin_worley = remap(perlin, 1.0-worley, 1.0, 0.0, 1.0); base_cloud = remap(low_freq_noise, high_freq_noise, 1.0, 0.0, 1.0) — remap instead of multiply 'prevents a loss of too much density at the core'; cloud_with_coverage = remap(noise, cloud_coverage, 1.0, 0.0, 1.0) — coverage as erosion makes clouds inflate/contract smoothly as coverage animates.

## 2017 anvil + wind/skew authoring: anvils are a pow() on coverage near layer top, gated by weather-state anvil_bias; coverage field is static, only noise scrolls [confirmed]
Exact anvil formula (2017 deck pp. 46-50): coverage = pow(coverage, remap(height, 0.7, 0.8, 1.0, lerp(1.0, 0.5, anvil_bias))). Raising coverage to a power < 1 above 70-80% of layer height inflates the coverage signal there, spreading cumulus tops into an anvil; anvil_bias (0-1, from the weather state) blends the exponent from 1.0 (no anvil) to 0.5. Anvils are thus 'a variation on the cumulus form', not a separate type — and a wind-vector variation skews anvils directionally for drama. Wind animation: p += wind_direction * time_offset; plus vertical shear skew p += height_fraction * wind_direction * 500.0 (500 m top-of-layer offset). Key 2017 deviation from 2015: 'while the noise in the density model is animated, the coverage signal is not' — structure stays put (art-directable), only detail evolves ('evolution of details, not cloud formations'). HZD shipped with only the low cloud layer volumetric; alto/cirro remained a 2D texture lookup.

## 2022 Nubis Evolved Vertical Profile Model: five 16km 2D NDFs — Min Height, Max Height, Coverage, Bottom Type, Top Type — with type channels indexing vertical-profile LUT textures [confirmed]
For HFW skies the weather map generalizes into a stack of 2D 'Nubis Data Fields' (NDFs) mapped over a 16 km × 16 km area, cloud layer roughly 256 m - 2048 m (slide axis labels). The five NDFs: Cloud Min Height and Cloud Max Height (per-column vertical range — this alone enables coexisting layers/towers of different heights), Cloud Coverage, Cloud Bottom Type and Cloud Top Type. Per Nubis Cubed's recap (p.19): 'Top Type and Bottom Type were used to sample vertical profile lookup textures and the results were multiplied to create what we call the vertical profile. Then we scaled this vertical profile by the cloud coverage data': float dimensional_profile = vertical_profile * cloud_coverage. The Top Type LUT axis is labeled Stratus->Stratocumulus->Cumulus over 0..1 (2015's analytic presets became texture LUT rows — trivially extensible to more genera by painting new LUT rows). Splitting bottom vs top type decouples base character from top character (flat sheared base + cauliflower top, or anvil top over cumulus body) and is Guerrilla's answer to authoring towering forms in a heightfield-ish model. Density: cloud_density = saturate(cloud_noise_composite - (1.0 - dimensional_profile)) using the same 4-channel 128^3 Perlin-Worley/Worley noise; animation is still noise scroll (sample_position - wind_direction * scroll_offset). Marcher: distance-adaptive step step_size = 3.0 + (60.0 * distance_from_camera) / 16384.0; temporal upscaling amortized over 16 frames (20ms->2ms) — fine for distant skies, unusable near camera. Multiple scattering volume: ms_volume = Remap(dimensional_profile*step_size, 0.1, 1.0, 0.0, 1.0) * pow(cloud_coverage * cloud_type, 0.25), then attenuated by pow(attenuated_light, cMultipleScatteringDepthPower) and pow(height_fraction, cMultipleScatteringHeightPower). Ambient: ambient_scattering = pow(1.0 - dimensional_profile, 0.5).

## 2022 cirrus/high-cloud solution: a 2.5D model where a SINGLE type scalar blends three baked cloud-texture channels (streaky/wispy/round) [confirmed]
The cirrus sub-layer (above the volumetric band) uses a '2.5D NDF Definition' of just Cloud Coverage + Cloud Type, and its density formula shows the type-transition pattern applied to 2D content: float density = ValueRemap(cloud_type, 0.5, 1.0, ValueRemap(cloud_type, 0.0, 0.5, cr_streaky, cr_wispy), cr_round); density = pow(density, 1.0 - ValueRemap(cloud_coverage, 0.0, 1.0, -0.9, 0.9)); density *= ValueRemap(pow(cloud_coverage, 3.0), 0.0, 0.5, 0.0, 1.0). cr_streaky / cr_wispy / cr_round are three channels of a baked cirrus texture; type 0->0.5 crossfades streaky->wispy, 0.5->1 fades toward round (altocumulus-like puffs), and coverage both sharpens (pow) and gates the result. This is directly relevant to Stellar Nomad R8: the same coverage/type fields drive a cheap 2D layer with per-type texture content, keeping far-field and volumetric consistent. There is also a 'Stratus Sub-Layer' fog band below the main volumetric layer.
ANCHORS: https://d3d3g8mu99pzk9.cloudfront.net/AndrewSchneider/NubisEvolved/NubisEvolved-NoVideos.pdf

## 2022 Envelope Model (fly-through orographic clouds): NDFs = Min/Max Height, Type, Density; type sets the HEIGHT at which detail flips wispy->billowy [confirmed]
For clouds the player flies through in HFW: dimensional profile is geometric, not weather-driven — float height_fraction = Remap(height, min_height, max_height, 0, 1); bottom_gradient = pow(height_fraction, 2.0); top_gradient = pow(1.0 - height_fraction, 1.5); edge_gradient = Remap(sample_height, 0.0, 35.0, 1.0, 0.0); dimensional_profile = product of the three. Detail-type semantics differ from the sky model: float noise_height_blend = Remap(height_fraction, cloud_type + 0.1, cloud_type - 0.1); composite = lerp(wispy_noise, billowy_noise, noise_height_blend) — i.e. cloud_type (0.25/0.5/0.75 examples shown) is the normalized altitude where wisps give way to billows within the cloud, matching real cumulus (wispy bases, billowy tops). Density: cloud_density_sample = height_fraction * pow(saturate(noise_composite - (1.0 - dimensional_profile)), 0.27). Sampling used cone-step mapping + distance-step mapping over height data (4.2ms -> 1.3ms), a two-pass resolution split near/far instead of temporal upscaling, and direct light = exp(-summed) * long-distance shadow-map sample. Envelope NDF also had placement fields (Cloud Distance, Upper Angle, Lower Angle). Limitations table (Nubis Cubed p.53): Evolution 'Pseudomotion Only', not visually intuitive to author — motivating voxels.

## 2022 authoring/composition: NDF Generator + NDF Editor + Influence NDFs; weather states select cloudscape NDF stacks per region — closest public analogue to Star Citizen-style map-stack planets [confirmed]
Authored NDFs come from three tools: a procedural NDF Generator, a Houdini NDF Editor, and 'Influence NDFs' — masked overrides blended into the procedural fields (slide: 'Vertical Profile Model NDF Definition' + Influence Mask + '2.5D NDF Definition' + Influence Mask = 'Influence NDF Definition'). Examples: 'Cumulus Procedural NDFs + Regional Influence NDFs', San Francisco region with 'Stratocumulus Procedural NDFs + Regional Influence NDFs'. Weather states map to whole cloudscape definitions: 'Calm Cloudy Weather State -> Partly-Cloudy Cumulus Cloudscape + Regional Influence NDFs; Stormy Weather State -> Stormy Cumulus Cloudscape; Calm Clear Weather State -> Light Cirrus Cloudscape'. Superstorms (cumulonimbus supercells) are authored the same way: 'Superstorm Influence NDFs' = Min Height, Max Height, Coverage, Bottom Type, Top Type + a Superstorm Mask; the mesocyclone is animated with the 'M.D.R. Vortex Field' — concentric rings n=0..5 each with ring_rotation_speed[n] and ring_skew[n] rotating the noise sample position around the storm center (noise = SampleNoise(GetRotatedPosition(sample_position, superstorm_center, time_offset * ring_rotation_speed[n] + ring_skew[n]))), with matching reprojection-vector math so temporal upscaling doesn't smear; the anvil is a vertical gradient + a cirrus NDF radiating from the superstorm center; internal lightning glow: potential_energy = pow(1.0 - d1/radius, 12.0) * height_gradient * (1.0 - saturate(fine_density * 5.0)). PS4->PS5 scaling table: light ray samples 6->10, view ray samples 60-90 -> 96-180, max res 960x540 -> 1920x1080, total <=4ms (PS4) vs <=2-3ms (PS5).

## Nubis3 (2023) NVDF voxel clouds: 3D fields (Dimensional Profile from an SDF, Detail Type, Density Scale @ 512x512x64 BC6, ~16.8MB) + separate BC1-packed SDF; NO weather map for the flyable layer [confirmed]
For Burning Shores' flyable clouds the 2D map pipeline is replaced by artist-built voxel cloudscapes: fluid-sim pieces (custom Houdini 'Aero' solver) composited with the 'Atlas' voxel tools into 'frankencloudscapes' over the 4km × 4km × 500m DLC airspace. Storage: Nubis Volume Data Fields (NVDFs) at 512×512×64 (8 m precision; a raw density grid at 2m precision would have been 2048×2048×256 = 2.1GB uncompressed — rejected): (a) Dimensional Profile — 'generated from a signed-distance field of the cloud to ensure that we get a gradient from outside to inside', i.e. the depth-into-cloud probability field that density up-rez, ambient (pow(1-profile,0.5)) and multiple-scatter approximations all reuse; (b) Detail Type — wispy->billowy blend factor per voxel; (c) Density Scale — post-noise density multiplier; BC6, 3 channels, 1 byte/texel, 16.777 MB total. Separately, Field Data NVDF: a true SDF (range -256m..4096m rescaled to 0..1), 512×512×64, packed into BC1 via a custom compressor (5-6-5 bit-split across RGB, decoded with dot(rgb, float3(1.0, 0.03529415, 0.00069204)), always rounding DOWN so sphere tracing never oversteps) — 10-30% faster than uncompressed. Memory per cloudscape across methods: Vertical Profile 0.541MB, Envelope 9.437MB, Voxel 25.166MB. Weather/type maps do NOT drive this layer: structure is static ('Evolution: Pseudomotion Only' — wind-scrolled noise via cCloudWindOffset only), evolution + on-the-fly SDF generation named as future work; cinematics reuse the cloudscape via transforms; bespoke cloudscapes built per encounter (Stormbird storm cloud with interior cave, egg, lightning reveal, localized rainstorm from the HFW local weather system).

## Nubis3 detail-noise philosophy: Alligator (billows) + Curly-Alligator (wisps), lo/hi-freq blended over the dimensional profile, wispy-vs-billowy by Type, and 'twice-folded' noise for <150m fly-through detail [confirmed]
Physical rationale: 'layered Wisps and curly distortions form where density is decreasing and layered billows occur where density is increasing.' New 4-channel 128^3 texture (2 bytes/texel, 4.19MB): ch R/G = low/high-frequency Curly-Alligator (inverted SideFX Alligator noise distorted by curl noise — replaced Perlin-Worley because it 'was not purely wispy enough'), ch B/A = low/high-frequency Alligator (replaced inverted Worley: 'more appropriate cloud-like lacunarity' than packed spheres). Sampled once per density sample at inSamplePosition * 0.01 with MIP level rising over distance: mipmap_level = log2(1.0 + abs(distance * cVoxelFineDetailMipMapDistanceScale)) (~15% perf win). Up-rez composite (full HLSL on deck p.125): wispy_noise = lerp(noise.r, noise.g, dimensional_profile) — high-freq wisps at edges, structure inside; billowy_noise = lerp(noise.b*0.3, noise.a*0.3, pow(dimensional_profile, 0.25)); noise_composite = lerp(wispy_noise, billowy_noise, Type). Fly-through extra octave without extra fetches: hhf_noise = saturate(lerp(1.0 - pow(abs(abs(noise.g*2-1)*2-1), 4.0), pow(abs(abs(noise.a*2-1)*2-1), 2.0), Type)) ('twice folded noise'), blended in only within 50-150 m of camera (ValueRemap(dist, 50, 150, 0.9, 1.0)). Final density: uprezzed_density = ValueErosion(dimensional_profile, noise_composite) * pow(saturate(DensityScale),4), then sharpened: pow(uprezzed_density, lerp(0.3, 0.6, powered_density_scale)), plus a near-camera pow(., lerp(0.5,1.0,distblend)) * lerp(0.666,1.0,distblend) to add definition and cut undersampling noise. Net: 0.5 m apparent precision from 8 m voxels.
ANCHORS: https://d3d3g8mu99pzk9.cloudfront.net/AndrewSchneider/Nubis%20Cubed.pdf

## Nubis3 fly-through rendering: sphere-traced SDF march + sqrt-distance adaptive step + near/far jitter; decoupled voxel light grid (-40%); 2-pass resolution split; 2.2-4ms @960x540 PS5 [confirmed]
March (full HLSL on deck p.174): per step, adaptive_step_size = max(1.0, sqrt(distance)*0.08); cloud_distance = SDF lookup; step = max(cloud_distance, adaptive_step_size); jitter offset = (distance < 250 ? animated_hash : static_hash) * step (animated jitter near camera hides undersampling, static far avoids shimmer); density+light samples taken only when SDF < 0. Lighting decoupling: only the first 2 secondary-march samples toward the sun are taken live; the rest are replaced by a precomputed 256×256×32 grid of summed density toward the sun (for each cloud voxel, a ray is marched to grid bounds accumulating density), rebuilt amortized over 8 frames at 0.1-0.2 ms — cut total render ~40%, preserved crisp near-surface lighting, and added long-distance inter-cloud shadows previously impossible (old method: 256 m, 10-sample light march). Same grid summed toward the sky gives directional ambient: ambient_scattering = pow(1.0 - dimensional_profile, 0.5) * exp(-summed_ambient_density). Multiple scattering: ms_volume = dimensional_profile * exp(-inSunLightSummedDensitySamples * Remap(sun_dot, 0.0, 0.9, 0.25, ValueRemap(cloud_distance, -128.0, 0.0, 0.05, 0.25))) — relaxed near sun direction and deep inside cloud for inner glow. Perf @960x540 PS5 30Hz (base -> +voxel lighting -> +SDF/adaptive march): ground view 10 -> 6.1 -> 2.2 ms; mixed air view 12 -> 8.2 -> 4.0 ms; full immersion 10 -> 8.0 -> 4.0 ms; geometry pass cost falls as cloud cost rises (self-balancing budget). 40/60Hz: split render <200 m at 480×270, >200 m at 960×540 -> 4 ms becomes 2.1 ms with only minor pixelation on dense features. Freebies published: bit.ly/NubisVoxelCloudPack (two production NVDFs as TGA slices + VDB, the 4-channel noise + Houdini generator HDA); 2017's noise HDA at bit.ly/nubisnoisegen; Alligator noise source via SideFX HDK docs.

## How the lineage blends type transitions (answer to Stellar Nomad's binary-border problem): interpolate profile PARAMETERS by a continuous type scalar; never dither, never blend finished looks [confirmed]
Across all four talks there is no dithering and no discrete per-type rendering: (1) 2015/2017 — one type scalar slides remap in/out points of the height gradient continuously through stratus->stratocu->cumulus; every intermediate value is a valid profile, so spatial type variation reads as morphing cloud genera, not a seam. (2) Coverage and type are INDEPENDENT low-frequency fields, so mixed skies (stratus band beside cumulus columns) come from painting/generating spatial variation, not from a coverage->type coupling — Stellar Nomad's current cloudType=smoothstep(coverage) collapses this two-axis space to a line, which is precisely why it yields exactly two looks; 2017 slide 66 shows 'the reduction in the blue channel corresponds to a stratus cloud band' within one scene. (3) 2022 — type becomes LUT coordinates (Bottom/Top Type sampled in vertical-profile lookup textures, results multiplied), so adding genera = adding LUT content, and transitions are texture filtering between LUT rows; per-column Min/Max Height gives independent altitude authoring. (4) Anvil/cumulonimbus are modifiers on the cumulus end (pow on coverage near layer top with anvil_bias, or superstorm influence-NDF overrides + cirrus NDF anvil), not new branches. (5) Detail noise follows the same continuous axes: wispy<->billowy by lerp over type (2015/2017 via inverted Worley at bases; 2022 envelope: by height with type setting the crossover altitude; Nubis3: lerp(wispy,billowy,Type) with each composite internally blended lo->hi frequency over the dimensional profile). (6) Temporal weather transitions are plain lerps of state parameters over 10-30 s. Vertical layering handles coexisting genera at different altitudes: volumetric minus-layer + separate cirrus sub-layer (2D/2.5D) + stratus fog sub-layer, all reading the same/consistent field stack.

## RECOMMENDATIONS
- Replace Stellar Nomad's coverage-derived cloudType with an independent type channel (or two: bottomType/topType a la Nubis Evolved) in the weather texture; keep coverage as erosion via remap(noise, coverage, 1, 0, 1) so coverage animation inflates/deflates rather than pops.
- Implement type->profile as parameter interpolation, not profile crossfade: either the 2017 form (type slides remap in/out points across an ordered genus sequence) or, better for R1/R3/R4, the 2022 LUT form — a small 2D texture where u=alt01 and v=type samples the vertical profile; then vertical_profile = topTypeLUT * bottomTypeLUT, dimensional_profile = vertical_profile * coverage. New genera (cirrus, altocumulus, congestus, cumulonimbus) become painted LUT rows — cheap to iterate behind a build const, one extra fetch (or keep analytic remaps in TSL for zero fetches).
- Author anvils/cumulonimbus as modifiers on the cumulus end, not new types: coverage = pow(coverage, remap(alt01, 0.7, 0.8, 1.0, lerp(1.0, 0.5, anvil_bias))) with anvil_bias driven by a precipitation/CAPE-like channel; add a wind-shear skew p += alt01 * windDir * shearMeters.
- Adopt per-column cloudMinAlt/cloudMaxAlt channels (Nubis Evolved) instead of a single covSpan-gated topAlt — this is what allows coexisting low stratus next to towering congestus and is straightforward to bake from real data (cloud-top height/pressure products) for R3.
- For R8, note Guerrilla's split: the same coverage/type fields feed a 2.5D cirrus/far layer whose density formula (streaky/wispy/round texture channels blended by type, sharpened by coverage) is a good template for the far-field overlay re-bake.
- Blend detail noise by the same axes: noise_composite = lerp(wispy, billowy, type), with each composite lerping low->high frequency over depth-into-cloud (dimensional profile); consider Alligator/Curly-Alligator channels and the twice-folded abs-fold trick (zero extra fetches) for close fly-through detail within ~150 m, plus the near-camera density-sharpening pow to fight undersampling.
- For fly-through performance on M2 Pro, the highest-leverage Nubis3 levers in fetch-per-voxel currency are: (1) precomputed summed-density light volume (coarse, amortized over N frames) replacing most light-march samples (-40% for Guerrilla, also buys inter-cloud shadows), (2) step placement informed by emptiness (SDF or a cheap coarse occupancy/cone map) max()ed with sqrt(distance)-scaled adaptive steps, (3) near/far split-resolution rendering (~-50%).
- Weather-state machine (2017) maps cleanly to procedural planets (R4): per-climate-zone sets of {cloud map generator params + skew/anvil/density/wind scalars}, scheduler lerps states over tens of seconds; for gas giants (R5) the superstorm mesocyclone technique (concentric rotation rings with per-ring speed/skew rotating noise sample positions, plus matching reprojection vectors) is the published starting point for banded/vortex flow without simulation.
