# web:engines-gasgiants

## SUMMARY
Across shipped engines there are three dominant ways to parameterize cloud types, and all three avoid the binary-border failure mode Stellar Nomad currently has. (1) The Guerrilla/Nubis lineage (which the current earthClouds.ts descends from) uses a weather map with SEPARATE channels for coverage and type (SIGGRAPH 2015: R=coverage, G=precipitation, B=type 0..1 stratus->cumulus), and by 2017-2023 replaced the three analytic height gradients with artist-authored VERTICAL-PROFILE LOOKUP TEXTURES indexed by (type, normalizedAltitude), plus per-cell cloud min/max heights and independent 'bottom type' and 'top type' indices — a continuous type axis with LUT-stored profiles, not analytic curve mixing. (2) Epic's current UE5 cloud-system content goes further: the weather map's FOUR channels are four coexisting genera (R=stratocumulus, G=altostratus, B=cirrostratus, A=nimbostratus), each with its own per-channel height-profile texture; types coexist and blend because they are summed independent fields, not a single scalar that switches looks. (3) MSFS 2020/2024 has no type channel at all: meteoblue streams a gridded 3D atmosphere (hourly forecast grid, 60 vertical levels, cloud fraction + liquid/ice water content, wind, temperature, humidity per level) and genera EMERGE from where in altitude the cloud water sits and how convective the column is; its preset system exposes the same idea as up-to-3 independent layers each with base/top altitude, density and scattering. Frostbite (Hillaire 2016) and RDR2 (Bauer, SIGGRAPH 2019) both additionally split HIGH clouds (cirrus family) into a separate cheap layer — RDR2's realtime global cloud map is explicitly a two-channel texture = density of two layers, shaped by a table of height gradients.

The convergent lesson for R1/R2: (a) make coverage-per-genus (or coverage+continuous-type+top-height) the map payload rather than deriving type from coverage; (b) store vertical profiles in a small LUT texture indexed by (type, alt01) instead of exactly three analytic curves — this is one extra (highly cache-coherent, per-column-constant) fetch and makes new genera data-driven; (c) treat the cirrus/high family as a second thin layer that shares the same weather map (satisfies R8 naturally, and matches Frostbite/RDR2/MSFS2024 all adding cirrus as a distinct layer, not a type value); (d) treat cumulonimbus/anvil as an injected/placed anomaly (Guerrilla's HFW superstorm is a special asset driven by gameplay, not emergent weather-map values).

For gas giants (R5) there is essentially no published fly-through prior art with real detail: Space Engine renders gas giants as ~7 differentially-rotating textured shell layers with procedural band/cyclone parameters (no shipped 3D volumetrics), Elite Dangerous uses opaque procedurally-textured spheres you cannot enter, and No Man's Sky Worlds Part II (Jan 2025) ships descend-able gas giants 10x planet size with planet-wide storms but has published no rendering internals. The credible design path is to synthesize from planetary science plus the procedural-texture tradition: Jupiter's weather layer is three chemically-fixed decks (NH3 ice topping out around 0.7 bar, NH4SH near 2-2.5 bar, H2O based at 5-7.5 bar; ~25 km pressure scale height, so the whole weather layer spans roughly 80-150 km — conveniently similar in thickness to an Earth cloud slab, just wider and deeper), and its horizontal organization is a quasi-static 1D latitude structure (about 30 alternating zonal jets up to ~140 m/s bounding belts=dark/downwelling/cleared-NH3 vs zones=bright/upwelling/high NH3 tops) plus a sparse list of long-lived vortices (GRS ~14-16,000 km, tops ~8 km above surroundings) and shear turbulence. That maps cleanly onto your existing abstraction: replace the equirect coverage map with a per-latitude band table (deck-top altitude, per-deck clearing/'coverage', tint, turbulence amplitude) domain-warped by a zonal flow field and stamped by a vortex list, and reinterpret 'cloud type' as 'deck index' with per-deck vertical-profile LUT columns — the same (coverage, type/top-height, profile-LUT) consumption path the Earth volumetric and 2D far-field would use (R4/R5/R8).

## Nubis/HZD (SIGGRAPH 2015): weather map = R coverage, G precipitation, B cloud type; 3 height gradients blended by type [confirmed]
Andrew Schneider's 'The Real-Time Volumetric Cloudscapes of Horizon Zero Dawn' is the direct ancestor of Stellar Nomad's current system. Weather map channel semantics: R = coverage (0..1), G = precipitation (drives rain probability, pushes coverage up and transitions clouds toward towering rain clouds), B = cloud type (0 = stratus, 0.5 = stratocumulus, 1 = cumulus). Crucially, TYPE IS AN INDEPENDENT CHANNEL, not derived from coverage — that derivation (cloudType = smoothstep(coverage)) is Stellar Nomad's own shortcut and is the root cause of the two-looks-planet-wide/binary-border symptom. Vertical shape: three analytic height-gradient presets (each a 4-point remap over normalized altitude) linearly blended by the type value; cumulonimbus emerges when precipitation channel forces coverage+type high. Rendered <2 ms on PS4.

## Nubis2/Nubis3 evolution: analytic gradients replaced by vertical-profile LOOKUP TEXTURES indexed by (type, altitude), with separate bottom-type and top-type [confirmed]
Nubis 2017 ('Authoring Real-Time Volumetric Cloudscapes with the Decima Engine') introduced artist-authored 'dimensional/vertical profiles' replacing the three fixed gradients. By Nubis3 / 'Nubis, Cubed' (SIGGRAPH 2023, Horizon Burning Shores), the non-voxel 'vertical profile' cloudscape path is parameterized as: a set of 2D maps (~5) including cloud MIN and MAX heights defining the vertical range of the slab per texel, plus BOTTOM TYPE and TOP TYPE indices that each sample vertical-profile lookup textures — i.e., per-genus vertical density curves live in LUT texture columns, the map picks which column at cloud base and which at cloud top, and the marcher lerps between them over altitude. This gives continuous genus morphing (cumulus humilis -> congestus is just a type-index ramp) and independent bottom/top character (e.g., flat stratiform base with convective top). For fly-through, Nubis3 adds baked voxel clouds (NVC assets): voxel grids storing Dimensional Profile (0..1 gradient toward cloud surface), Detail Type (billow vs wispy blend), Density Scale; in-shader detail erosion from a 128^3 RGBA noise (R,G = low/high-freq curl-alligator -> billow; B,A = low/high-freq alligator -> wispy); compressed SDFs for empty-space skipping and a 256x256x32 light-accumulated-density voxel grid replacing per-sample light marches (claimed 30-40% render time saving). Design takeaways for R1/R2/R6: (1) type = continuous LUT index, profiles are DATA not code; (2) detail character (billow/wispy) is a separate axis from the density profile — matching your existing detail/lighting decoupling; (3) fly-through demanded baked/voxel + SDF acceleration, not just the 2D-map path.

## Guerrilla superstorms (GDC 2022): cumulonimbus/supercell handled as a special injected system, not emergent weather-map values [confirmed]
'The Real-Time Volumetric Superstorms of Horizon Forbidden West' (Schneider, GDC 2022): the tropospheric Nubis system was EXTENDED with dedicated superstorm elements — tornadic supercell shapes, fast cloud movement via temporal upscaling, internal lighting/lightning — driven by story/gameplay placement rather than by pushing the ambient weather map to extreme values, and scaled across PS4/PS5 without fluid sim. Takeaway for R1: rare, dramatic genera (cumulonimbus with anvil, supercells) are best parameterized as sparse placed/injected anomalies (position, radius, intensity, anvil spread) stamped into the same weather-map fields, rather than as a reachable corner of the global type axis — keeps the common-case marcher cost unchanged (R6) and gives art/gameplay control.

## UE5 official cloud-system content: four coexisting genera as RGBA weather-map channels, each with its own height-profile texture channel [confirmed]
Epic's current Volumetric Cloud material documentation (UE 5.7 'cloud system' plugin content) parameterizes the sky as FOUR cloud types aligned to texture channels: R = stratocumulus, G = altostratus, B = cirrostratus, A = nimbostratus (storm). Two paired textures: 'Layout_CloudGlobalPattern' (per-channel world-space coverage field for each genus) and 'Layout_CloudHeightProfile' (per-channel vertical density profile vs altitude for that genus, including its relative altitude band). Global controls: Layout_CloudType (per-type visibility weights), Layout_GlobalCoverage, Layout_CloudGlobalScale (repeat distance in km), Layout_CloudPerTypeScale, Layout_GlobalCloudMask + Layout_CloudTypeMask (local add/remove override). Because the four genera are independent summed fields, types coexist in the same sky and transitions are automatically smooth (crossfading coverage fields), directly solving the R2 'binary border' problem; and the four coverage values cost ONE RGBA fetch — type multiplicity is fetch-neutral (R6). The older engine path (Volumetric Cloud component + m_SimpleVolumetricCloud / Volumetrics plugin m_VolumetricAdvanced_TwoLayers) supports one component with materials that evaluate up to two layers; profiles are material-graph-defined (paintable weather map + conservative density). SkyAtmosphere coupling (aerial perspective applied to clouds, sun transmittance sampled from atmosphere LUTs) exists but is not detailed on this page.
ANCHORS: https://dev.epicgames.com/documentation/unreal-engine/volumetric-cloud-material-in-unreal-engine | https://dev.epicgames.com/documentation/en-us/unreal-engine/volumetric-cloud-component-in-unreal-engine

## MSFS 2020/2024: no type channel — genera emerge from a streamed 3D gridded atmosphere (meteoblue, 60 vertical levels, cloud fraction + water content per level) [confirmed]
meteoblue supplies MSFS hourly gridded forecasts: horizontal grid (~250M cells marketing figure, multi-resolution NEMS model blend) x 60 VERTICAL LEVELS from surface to stratosphere; per-cell fields include wind, temperature, humidity, CLOUD FRACTION, liquid/ice water content, precipitation, pressure, aerosols. Asobo interpolates this spatially+vertically into a continuous 3D atmosphere and the volumetric renderer samples it: cumuliform vs stratiform looks EMERGE from the vertical distribution (thin wide fraction at one level = stratiform; deep moist convective column = towering cumulus), precipitation emerges from moisture + vertical development. The authored/preset system (SDK 'Weather Definitions') exposes the same abstraction as up to THREE independent CloudLayer elements, each: CloudLayerAltitudeBot, CloudLayerAltitudeTop (meters), CloudLayerDensity (0..1), CloudLayerScattering (0..1), plus coverage octas in UI. MSFS 2024 added new cloud types, notably CIRRUS as a distinct high-atmosphere layer, plus denser storms. Takeaway for R3/R4: the 'real data' representation used by the most weather-serious sim is (per-level cloud fraction + water content), i.e., LAYERED SCALAR FIELDS, not genus labels — genus is a derived presentation. A 3-4 layer bake (low/mid/high coverage + convective potential/top-height) is the faithful compression of it.

## Frostbite (Hillaire SIGGRAPH 2016): Schneider-style coverage/type weather texture for the volumetric slab + a SEPARATE cheap 2D high-cloud (cirrus) layer [likely]
Hillaire's 'Physically Based Sky, Atmosphere and Cloud Rendering in Frostbite' couples the volumetric low/mid cloud slab (weather-texture-driven, coverage+type parameterization following Schneider 2015) with a separate high-altitude cloud layer rendered as cheap panning 2D textures (cirrus/cirrostratus family), both lit consistently by the same atmosphere LUTs (sun transmittance at cloud altitude, aerial perspective on top). Frostbite's other main contributions are the energy-conserving analytic scattering integration per raymarch step and sky/atmosphere unification — which Stellar Nomad already implements. The two-tier split (volumetric troposphere slab + thin 2D-ish cirrus deck) is the industry-standard answer to 'where does cirrus come from' — no shipped engine tries to raymarch cirrus in the same slab as cumulus. Confidence 'likely' on exact channel semantics because the course PDF exceeded fetch limits; the layered-cirrus architecture is consistently reported by multiple implementation writeups.

## RDR2 (SIGGRAPH 2019, Bauer): realtime global cloud map = TWO channels = density of TWO layers, shaped by a table of height gradients [likely]
'Creating the Atmospheric World of Red Dead Redemption 2: A Complete and Integrated Solution' (Advances in Real-Time Rendering, SIGGRAPH 2019): the engine generates, in real time and covering the whole world, a cloud map as a TWO-CHANNEL texture holding cloud density for TWO layers (low volumetric layer + high layer), as a function of weather state and time of day; a TABLE OF GRADIENTS (height-density curves) produces the different cumulus vs layered (stratiform) looks; cirrus/high clouds ride the second channel/layer. It also unifies clouds, fog and sky into one voxelized raymarch + irradiance model for directability. This is the minimal-cost multi-type pattern: 2 channels + N stored gradient curves selected per weather state — a good existence proof that even 'coverage-per-layer x gradient-table' (without any explicit type channel) already yields multiple believable genera without binary borders.

## Space Engine gas giants: ~7 differentially-rotating textured shell layers + procedural stripe/cyclone parameters; shipped volumetrics are NOT in gas giants yet [likely]
SpaceEngine's own dev blog ('Multi-layered gas giants'): gas giants are rendered as 7 CLOUD LAYERS, all rotating around the planet at different speeds so counter-flowing jets appear at the boundaries; atmosphere via Bruneton scattering (noted unstable at high haze density). The procedural texture generator exposes per-planet parameters for banding and storms — stripe zone count/fluctuation/twist and cyclone magnitude/frequency/density/octaves in the .sc planet-script format (i.e., bands = 1D latitude function with noise fluctuation + twist shear; cyclones = injected vortex features) [this parameter-name detail is from training knowledge of SE's .sc format: likely]. SE 0.990+ has raymarched volumetric nebulae and (post-Vulkan) plans 3D volumetric clouds, but as of available material, flying 'into' an SE gas giant just descends through alpha-blended shells into haze — there is no published fly-through volumetric gas-giant renderer. Elite Dangerous likewise renders gas giants as opaque procedurally-banded textured spheres with no atmospheric entry (likely, from community/graphics-mod material; Frontier has published nothing technical).

## Real Jupiter vertical structure: three chemically-fixed decks (NH3 ~0.7 bar, NH4SH ~2-2.5 bar, H2O base ~5-7.5 bar), ~25 km scale height, whole weather layer ~100-150 km thick [confirmed]
Consensus structure from NASA/ScienceDirect/arXiv sources: NH3-ice deck condensing above ~0.7 bar (tops pushed to ~0.2-0.5 bar in vigorous zones), NH4SH (ammonium hydrosulfide) deck above ~2.5 bar, H2O deck (liquid base, frozen tops) based ~5-7.5 bar (some models put aqueous-ammonia base at 8 bar); observed NH3 ice particle scale height ~30% of the ~25 km gas scale height (i.e., decks are THIN, ~7-10 km each, like Earth stratiform sheets). Converting with H ~= 25 km (T~165 K at 1 bar, mu~2.2, g~24.8): NH3 deck ~ +9 to +40 km above the 1-bar level, NH4SH ~ -18 to -25 km, H2O ~ -40 to -50 km — total flyable weather layer ~100-150 km deep, i.e., only ~10x your current Earth slab, NOT thousands of km; below the H2O deck it is clear hot haze thickening forever (natural fog-wall bottom, no surface needed). Saturn: same deck sequence but colder so everything sits 2-3x deeper, ~60 km scale height, overlaid by thick photochemical haze (blander look). Parameterization implication (R5): 'type' on a gas giant is DECK INDEX with globally-fixed altitudes; the weather map only needs per-deck clearing/thickness and top-altitude perturbation, so a (deckClear[3], topBoost, tint) texel is sufficient.

## Real Jupiter horizontal structure: quasi-static 1D latitude skeleton (jets/belts/zones) + sparse long-lived vortices + shear turbulence — ideal for a band-table parameterization [likely]
Jupiter's horizontal organization is dominated by ~30 alternating east-west zonal jets (peak ~140-150 m/s at the 23-24N jet; Saturn's equatorial jet reaches ~370-470 m/s), stable over decades — so a STATIC per-planet 1D latitude function is physically faithful, with time evolution being pure zonal advection (differential rotation: lon += u(lat) * t / (R cos lat)). Belts = dark, warmer, downwelling, NH3 deck thinned/cleared exposing brown/orange chromophores of the deeper decks; zones = bright, cold, upwelling, thick high NH3 ice tops. Vortices: Great Red Spot (anticyclone, currently ~14-16,000 km across and shrinking, cloud tops ~8 km ABOVE surroundings, ~6-day rotation, turbulent wake downstream), white ovals (anticyclones in zones), brown barges (cyclones in belts), 5-um hot spots (cloud-free holes at the NEB southern edge), and Juno-discovered polygonal polar cyclone clusters (8 around 1 at the north pole, 5-6 south). Kelvin-Helmholtz billows and chevron/festoon turbulence concentrate at jet shear lines between bands. Numbers are well-established planetary science (training-verified against the anchored review paper).

## Procedural gas-giant band rendering tradition: latitude-stretched fBm + domain warp + smoothstep color decisions; bake noise to textures for perf [confirmed]
The published procedural approach (Paleologue's writeup, representative of SE/Elite-class shaders): sample 3D simplex/fBm at the sphere point with the latitude axis scaled (e.g., samplePoint.y *= 2.5) to squash noise into horizontal bands; latitude = warped y where warp = fractalSimplex(point, 5 octaves, lacunarity 2, gain 2) * warpStrength (~2.0); two offset warped samples feed smoothstep color decisions (smoothstep(0.4,0.6,d1), smoothstep(0.2,0.8,d2)) between three HSV-derived colors (primary hue ~240 +/- 30, complementary secondary, dark low-value band color). Perf note from the same source: real-time 3D noise is the bottleneck — bake to textures per seed and sample in-shader (for Stellar Nomad: bake the band/warp field into an equirect weather texture exactly like Earth's, satisfying R4's same-consumption-path requirement). Missing from all public work: vortices and flow animation — these must come from the vortex-list + zonal-advection design (see Jupiter findings); curl-noise/flow-noise advection of the warp field is the standard technique for animating shear turbulence.

## Cross-engine synthesis: the three viable payload schemas for Stellar Nomad's redesigned weather map, and their per-voxel fetch cost [likely]
Schema A (UE5-style, 'coverage-per-genus'): RGBA = coverage of 4 genus groups (e.g., R=stratus/stratocu low, G=cumulus congestus/convective, B=altocumulus mid, A=cirrus high); each genus has a vertical-profile column in a small (type x alt01) LUT; density = sum_i cov_i * profileLUT(i, alt01) * noise. Cost: 1 RGBA weather fetch (already paid today) + profile evaluation — the LUT can be one 64x64 R8 texture fetched 2-4x per sample, OR sampled once per COLUMN at march start since it varies only with altitude (near-free amortized). Coexistence and smooth transitions are structural (independent summed fields). Schema B (Nubis3-style, 'coverage + continuous type + minmax'): RGBA = coverage, bottomType, topType, topAlt; profile = lerp(profileLUT(bottomType, alt01), profileLUT(topType, alt01), alt01); most compact, continuous genus morphing, but simultaneous unrelated types at one texel (cirrus over cumulus) still needs a second layer. Schema C (MSFS/RDR2-style, 'per-layer density'): N layers x (coverage, base, top) with genus emerging from layer stats; best fit for real-data baking (R3) and trivially maps to a gas giant's decks. Recommended hybrid: Schema A channels for the tropospheric slab + one extra cheap high-layer (cirrus, 2D or thin-volumetric) + Schema C semantics when baking from real datasets; gas giants consume the SAME shader path with genus channels reinterpreted as deck-clearing channels and the profile LUT swapped per planet (R4/R5/R8). All schemas keep marcher fetches flat vs today because weather-map and profile-LUT reads are per-column, not per-voxel (R6).

## RECOMMENDATIONS
- Replace cloudType = smoothstep(coverage) with an independent type axis in the weather texture. Every shipped system stores type (or per-genus coverage) as its own channel; deriving type from coverage is the direct cause of the two-looks/binary-border symptom (R1/R2).
- Adopt per-genus coverage channels (UE5 pattern) for the tropospheric slab: pack 4 genus-group coverages into the existing RGBA weather fetch (low-stratiform, convective, mid, high). Density = sum of cov_i * profile_i(alt01). This is fetch-neutral for the marcher (R6) and makes type transitions structural crossfades (R2).
- Move cloudHeightProfile from 3 hardcoded analytic curves to a small (type x alt01) vertical-profile LUT texture (e.g., 64x64 R8, one column per genus, Nubis3 pattern). Sample it per-column (once per ray inside the slab entry), not per-voxel. New genera (cumulus humilis vs congestus, altocumulus) become rows of data behind a build const (R1/R7).
- Handle the cirrus family as a separate thin high layer (Frostbite/RDR2/MSFS2024 pattern) driven by the SAME weather texture channel — 2D-textured or 1-2 march samples — never inside the main slab march. This also gives the 2D far-field overlay and volumetric a shared source trivially (R8).
- Treat cumulonimbus+anvil and gas-giant vortices (GRS) as a sparse injected anomaly list (position, radius, intensity, top-boost, anvil spread) stamped into the weather fields at bake/update time (Guerrilla superstorm + Juno vortex pattern), not as an extreme corner of the global type axis (R1/R5).
- For real-Earth data (R3), bake to the per-genus channels from layered fields (per-level cloud fraction is how MSFS/meteoblue represents truth): low/mid/high cloud cover + convective proxy -> 4 channels. This same 'layered scalar fields -> genus channels' bake is exactly how procedural planets should generate weather too (R4): climate sim outputs humidity/instability per level, same baker runs.
- Gas giants (R5): parameterize as (a) a per-latitude band table (16-32 bands: jet velocity, per-deck clearing, deck-top altitude offset, tint, turbulence amplitude), (b) an equirect weather texture BAKED from that table + domain-warped fBm (latitude-squashed, warpStrength ~2) + vortex list, consumed by the same marcher; (c) vertical structure = 2-3 thin decks (NH3/NH4SH/H2O ~7-10 km thick each) spread over a ~100-150 km slab with per-deck profile LUT columns; (d) animate via differential zonal advection (lon += u(lat)*t/(R cos lat)) and curl-noise shear at band boundaries; (e) no surface: exponential haze density below the H2O deck creates a natural fog-wall bottom.
- Do NOT look for shipped fly-through gas-giant volumetrics to copy — none are published (Space Engine = rotating textured shells, Elite = opaque spheres, NMS = shipped but undocumented). Validate the deck-slab approach with your existing DEBUG_VIZ/falsification workflow per docs/CLOUD_DEBUGGING_LESSONS.md before committing to it (R7).
